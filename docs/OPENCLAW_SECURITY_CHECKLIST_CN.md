# OpenClaw 上线前安全检查清单（V1）

> 用于 `openclaw-security-audit-lite` 的 lead magnet / 自查清单 / 首批体检前置材料。

## 一、暴露面
1. 是否存在不必要的公网开放端口
2. OpenClaw 是否直接暴露在公网且未经过额外访问控制
3. 是否为管理入口设置了最小可达范围
4. 是否区分了测试环境与真实环境的访问边界

## 二、权限与执行边界
5. 是否明确限制了本地文件读写范围
6. 是否明确限制了 shell / exec / 节点执行能力
7. 是否检查过高权限工具是否真的业务必需
8. 是否避免让普通使用者直接获得高风险动作能力
9. 是否梳理过不同 agent / session 的权限边界

## 三、消息通道与外发能力
10. 是否梳理 Telegram / Feishu / WhatsApp / Email 等通道接入清单
11. 是否确认消息发送能力不会被误触发到真实用户/群组
12. 是否检查过第三方账号 token / webhook 的存放方式
13. 是否确认敏感环境不会把内部信息直接外发到外部渠道

## 四、技能与供应链
14. 是否记录当前启用的 skills / extensions 来源
15. 是否确认安装的第三方技能可信、可追溯
16. 是否避免在未知来源脚本/技能上直接给高权限
17. 是否保留技能安装、更新、启用的变更记录

## 五、日志、留痕与可追溯性
18. 是否保留关键操作日志
19. 是否能追溯谁在什么时间触发了关键动作
20. 是否能看到失败任务、异常调用、关键报错
21. 是否保留必要的审计信息而不是只有零散控制台输出

## 六、节点与多实例
22. 是否梳理当前在线节点和可执行节点清单
23. 是否确认节点权限没有超出其业务需要
24. 是否区分个人节点、测试节点、生产节点
25. 是否对多节点接入做了最小权限与最小暴露原则检查

## 七、成本与异常
26. 是否知道最近 7 天主要成本消耗来自哪里
27. 是否知道哪些任务/agent/skill 最容易失败
28. 是否设置了基础异常提醒或人工巡检机制

## 八、上线决策
29. 是否有上线前检查结论：可上线 / 需整改 / 禁止上线
30. 是否有明确整改优先级：先修什么、后修什么

## 快速判定建议
- **0-5 项不满足**：基础可控，但建议补齐报告与巡检
- **6-10 项不满足**：存在明显风险，建议先做一次完整体检
- **10 项以上不满足**：不建议直接进入真实环境

## 最值得先修的 3 类问题
1. **公网暴露 + 缺少额外访问控制**
2. **关键入口缺少鉴权或权限边界过宽**
3. **技能来源不明 + 缺少审计留痕**

如果这三类问题同时存在，不建议直接进入真实环境。
